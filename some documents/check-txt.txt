　　　　　　　　　　　　　　Abstract
The increasing reliance on digital systems and online services has made cybersecurity a critical concern for organizations worldwide. Traditional security testing methods, such as periodic audits and penetration tests, are often insufficient to detect all vulnerabilities due to limited scope, time constraints, and cost. This project presents the design and development of Crowdsourced Cybersecurity Platform with Simulation, PTaaS, and Advanced Matching, a secure and structured environment that connects organizations with ethical hackers to identify and remediate software vulnerabilities efficiently.
The platform allows organizations to define program scope, manage vulnerability submissions, and reward valid security findings. Ethical hackers can safely report vulnerabilities, receive recognition, and earn monetary rewards, all within a legally protected framework. By leveraging crowdsourced security expertise, the platform enhances the security posture of participating organizations while fostering a collaborative cybersecurity community.
Key features of the platform include vulnerability reporting and tracking, triage and validation mechanisms, secure communication channels, reward management, reputation systems for researchers, expert code review, SSDLC integrations with tools like Jira and GitHub, and live hacking events. This project demonstrates how crowdsourced cybersecurity platforms  reduce the risk of cyberattacks, and promote ethical hacking practices.
In addition to real-world crowdsourced vulnerability discovery, the platform integrates a Bug Bounty Simulation environment for learning purposes. The ssimulation mirrors real bug bounty workflows using simulated targets and data, allowing beginner, intermediate, and advanced users to practice vulnerability discovery and reporting. This environment operates independently from real programs and does not involve real organizations, production assets, or monetary rewards.

　　　　1.1.6 Bug Bounty Simulation Environment
Bug bounty platforms require practical experience in identifying, reporting, and validating vulnerabilities. However, beginners often face barriers such as legal risks, lack of confidence, and insufficient exposure to real-world workflows.
To address this challenge, the proposed system includes a Bug Bounty Simulation environment that replicates real bug bounty program features using simulated targets and controlled data. The simulation allows users to safely practice vulnerability discovery, structured reporting, severity classification, and communication workflows without impacting real organizations or systems.
The simulation environment supports multiple difficulty levels beginner, intermediate, and advanced to progressively build user competence. This approach enhances report quality, prepares researchers for real bug bounty participation, and strengthens the overall effectiveness of the platform.


　　　
　　　　　1.2  Statement of the problem
Many organizations, especially smaller ones, struggle with identifying security vulnerabilities due to limited resources for internal testing or expensive external audits.
Existing vulnerability discovery approaches suffer from several limitations:
High cost of traditional penetration testing.
Limited security expertise availability.
Lack of continuous security testing.
Poor coordination between organizations and ethical hackers.
Limited local platforms supporting regional cybersecurity communities.
Lack of safe environments for beginners to practice ethical hacking.
Poor report quality due to limited hands-on experience.
Legal and ethical risks for inexperienced researchers.
 Absence of structured learning paths linked to real bug bounty workflows.
These challenges expose organizations to undetected vulnerabilities, data breaches, and financial loss.

Requirement Modeling Approach
After gathering requirements, they were structured and modeled using an Object-Oriented (OO) approach. 
The Object-Oriented approach was selected over the structured approach because
The system involves multiple interacting entities such as Hacker, Organization, Program, BugReport, and Reward
OO modeling supports modularity, scalability, and maintainability
It aligns with modern web application development and commercialization goals
Structured modeling was considered less suitable due to the dynamic and evolving nature of crowdsourced security platforms.
　　　1.4.2 Analysis and Design Methodology
The project adopts an Object-Oriented Analysis and Design (OOAD) methodology rather than a structured analysis approach.
The Object-Oriented methodology was selected because the Crowdsourced Bug Bounty Platform is composed of several interacting real-world entities such as Ethical Hacker, Organization and Staff. These entities possess attributes, behaviors, and relationships that are naturally and efficiently represented using object-oriented concepts.
The OOAD approach provides the following advantages
Modularity: The system is divided into well-defined classes, making it easier to manage and understand.
Scalability: Supports future expansion such as mobile applications, automation, and regional commercialization.
Maintainability: System changes can be implemented with minimal impact on existing components.
Reusability: Common functionalities can be reused through inheritance and composition.
Industry Relevance: OOAD aligns with modern web-based and enterprise security system development practices.
The structured analysis approach was not selected because it focuses primarily on functional decomposition and process flows, which are less suitable for systems with complex interactions and evolving requirements such as crowdsourced security platforms.

The following challenges and constraints have hindered the full achievement of the planned objectives
Limited Development Time: As a final-year undergraduate project with a fixed academic timeline, only the core functionalities could be analyzed and modeled in detail. Full implementation, testing, and deployment of a production-grade system were may not possible.
Full AI-driven BountyMatch not implemented due to time limits instead basic rule-based matching used.
Resource Constraints: Lack of access to commercial-grade infrastructure (e.g., high-availability servers, advanced security auditing tools) and budget for third-party integrations e.g real payment gateways.
Scope Prioritization: Focus was placed on requirement analysis and design (as required by the RAD), rather than  deployment and post-launch maintenance.
Regulatory and Compliance Challenges: Detailed handling of Ethiopian financial regulations, tax reporting for bounty payouts, could not be fully incorporated due to  time limits.
Activities left undone due to these constraints
Full implementation and coding of the system.
Integration with real payment gateways (e.g., Ethiopian banks, Telebirr, CBE Birr).
Mobile application development (native iOS/Android apps).
Advanced AI-based report triaging or researcher matching.
 Complete legal compliance module .
　　　　　　　　1.8 Scope of the Project
The scope of the Crowdsourced Bug Bounty Platform is defined to closely align with system requirements and bussiness logic, focusing on managed bug bounty programs while incorporating elements of vulnerability disclosure and researcher engagement. 
The system boundaries include the following business processes and subsystems.
I.User and Role Management Subsystem
Support for multiple roles: Organizations (customers), Ethical Hackers/Researchers (the Crowd), and Administrators/Platform Managers and staff.
Registration, profile management, authentication, and researcher vetting/profile enrichment.
II.Program Creation and Management Subsystem
Organizations can launch and manage public, private (invite-only), or managed bug bounty programs.
Define scope (in-scope/out-of-scope assets), rules, reward structures (tiered bounties based on severity), and program briefs.
Support for Vulnerability Disclosure Programs (VDP) as non-rewarded intake channels.
III.Researcher Engagement and BountyMatch Subsystem
Researchers can browse, join, and participate in programs. 
Advanced matching of researchers to programs and PTaaS requests using algorithms based on skills, vulnerability history, and performance metrics.
IV.Vulnerability Submission and Reporting Subsystem
Researchers submit detailed reports with evidence, reproduction steps, impact assessment, and attachments.
Structured submission workflows compliant with Vulnerability Rating Taxonomy (VRT)-like severity scoring.
V.Managed Triage and Validation Subsystem
Platform-managed triage: Intake, validation, duplication checking, prioritization, and remediation advice.
Status tracking (New → Triaged → To Review → Resolved) with notifications.
VI.Collaboration and Communication Subsystem
Secure in-platform messaging and comments between researchers, organizations, and platform triage team.
Real-time updates and activity feeds.
VII.Reward and Payout Management Subsystem
Bounty awarding based on validity and severity.
Payout tracking and researcher reputation/leaderboard system.
VIII.Analytics, Reporting, and Dashboard Subsystem
Role-based dashboards for organizations (program insights, KPIs, vulnerability trends) and researchers (submissions, earnings).
Basic reporting and benchmarking.
IX.Administration and Integration Subsystem
Admin tools for program moderation, user management, and platform oversight.
Conceptual support for API integrations and workflow connectors (e.g., to SDLC tools).
X.Bug Bounty Simulation (Learning Environment) Subsystem
Controlled learning environment using intentionally vulnerable applications.
 Feature parity with real bug bounty workflows using simulated data.
Beginner, intermediate, and advanced difficulty levels.
 Fully isolated with no real-world impact.
No real organizations or monetary rewards.
XI.PTaaS Subsystem
On-demand, time-bound penetration testing engagements leveraging the platform's ethical hacker community via BountyMatch for precise researcher assignment.
Structured methodology-driven testing (e.g., OWASP, PTES) with defined scopes, timelines, and compliance-ready deliverables.
Real-time dashboards providing visibility into testing progress, methodology checklists, and emerging findings.
Integration with BountyMatch for intelligent matching of vetted researchers based on skills, reputation, and expertise.
Structured vulnerability reporting with validation, prioritization, remediation guidance, and free retesting for confirmed fixes.
Fixed or subscription-based pricing models with platform commission, separate from variable bug bounty rewards.
Fully integrated yet isolated workflows from ongoing bug bounty programs to ensure controlled access and no overlap.
XII.Live Hacking Events subsystem
Time-bound collaborative events as a promotional feature, with real-time dashboards and integration to standard report flows.
XIII.SSDLC Integration subsytem
API-based connections to Jira and GitHub for  vulnerability tracking in development pipelines.
Expert Code Review Crowdsourced code audits integrated with PTaaS, supporting static analysis and human review for source code security.
XIV.AI Red Teaming Subsystem

Organizations can create AI Red Teaming engagements targeting AI systems (LLMs, AI agents, ML models).
Specialized vulnerability reporting for AI-specific attacks (prompt injection, jailbreak, data poisoning, model extraction).
Integration with BountyMatch for matching researchers with AI/ML security expertise.
Compliance-ready reporting with security/safety/trust taxonomy.
Strict ethical guidelines and isolation from regular bug bounty programs.
   Out of Scope (will be included in the future) 
Attack Surface Management (automated asset discovery and risk ranking).
Advanced AI features .
Managed Vulnerability Disclosure at scale without bounties.

 System Features 
　　　　　2.1 The Existing System 
The Crowdsourced Bug Bounty Platform is a startup project currently in the planning and design phase, with no operational platform yet established. Therefore, there is no existing automated or semi-automated system within the startup organization itself. All activities related to vulnerability discovery and responsible disclosure that the future platform will manage are currently handled in a completely manual manner by ethical hackers and target organizations in Ethiopia.
The existing system refers to the prevailing practices in the Ethiopian cybersecurity ecosystem, where ethical hackers and organizations interact directly without any intermediary platform. These interactions are fully manual, unstructured, and depend entirely on personal initiative and communication channels.
　　　　2.1.1 How the Existing System Functions
The current process is 100% in person process, relying on email, phone calls, social media, or personal networks. There is no centralized portal, standardized forms, tracking system, or reward mechanism in place locally. Ethical hackers must individually identify and contact organizations, while organizations handle incoming reports on an ad-hoc basis with limited resources.The existing system operates as follows:
Organizations formally request security testing services from INSA or licensed private cybersecurity firms.
Security assessments are conducted on-site or through controlled engagements, often requiring physical presence, official letters, and long approval processes.
Testing is usually performed periodically, not continuously.
Reports are delivered as static documents (PDF or printed reports).
Vulnerability remediation is handled internally by the organization after receiving the report.
There is limited or no structured follow-up, real-time communication, or verification of fixes.
This approach is semi-automated and heavily dependent on institutional procedures.
　　　　　　　　　2.2  Proposed System
The proposed Crowdsourced Bug Bounty Platform is a secure, scalable, and fully automated web-based system designed as a commercial startup solution to directly address the problems identified in the existing manual vulnerability disclosure practices in Ethiopia. It serves as a trusted, localized intermediary that connects Ethiopian organizations with ethical hackers, streamlining the entire lifecycle of vulnerability discovery, reporting, validation, remediation, and rewarding.
　　　　2.2.1 Bug Bounty Simulation (Learning Environment)
While the core of the proposed system is a real crowdsourced bug bounty platform, an additional Bug Bounty Simulation environment is integrated to support learning and skill development.
The Bug Bounty Simulation is a fully isolated, non-production environment that implements the functional features of learning-focused platforms. It mirrors real bug bounty workflows including simulated target listing, vulnerability discovery, structured vulnerability reporting, severity assessment, and feedback.
All targets, vulnerabilities, reports, and outcomes within the simulation are intentionally simulated and have no real-world impact. No real organizations, domains, or monetary rewards are involved.
The simulation environment is organized into three levels.
Beginner Level: Guided challenges with hints and learning support.
Intermediate Level: Realistic vulnerability scenarios with limited guidance.
Advanced Level: Complex vulnerability chains with no hints, closely resembling real bug bounty difficulty.
Participation in the simulation is optional and does not affect researcher eligibility, reputation, or rewards in real crowdsourced bug bounty programs. Its primary goal is to improve user readiness, report quality, and overall platform effectiveness.
　　　　　　2.2.2 Bug Bounty Simulation Module Architecture
The Bug Bounty Simulation module is designed as a fully isolated subsystem within the Crowdsourced Bug Bounty Platform. It shares the same user interface patterns, report workflows, and severity classification mechanisms as real bug bounty programs but operates exclusively on simulated assets and data.
The simulation module consists of the following components
Simulated Targets Repository – Contains intentionally vulnerable web applications and APIs. 
Simulation Report Engine – Enables structured vulnerability report submission identical to real programs.
 Feedback & Scoring Engine – Provides automated feedback and learning insights without affecting real reputation or rewards.
 Isolation Layer – Ensures strict separation between simulation data and real bug bounty data.
This architecture ensures feature parity while maintaining security, ethical boundaries, and academic integrity.




　　　Key Features and Solutions to Existing Problems
The proposed system eliminates the inefficiencies, risks, and barriers of the current manual approach by introducing the following automated and structured solutions.
Centralized Platform Access 
Replaces scattered email and social media contacts with a single, professional web portal where organizations and researchers can register, log in, and interact securely.
Structured Program Management 
Organizations can easily create and manage public, private (invite-only), or non-rewarded Vulnerability Disclosure Programs (VDPs). They define clear scope, rules, reward tiers, and severity guidelines solving the problem of unclear or non-existent disclosure policies.
Standardized Report Submission
Ethical hackers submit vulnerabilities using guided forms with required fields (title, description, steps to reproduce, impact, attachments). This replaces unstructured emails and ensures complete, high-quality reports.
Automated and Managed Triage 
Platform staff (or organization teams) use built-in tools to intake, deduplicate, validate, prioritize, and assign severity (using a VRT-like taxonomy). This eliminates manual forwarding, delays, and duplicate handling issues.
Secure Collaboration and Communication 
In-platform messaging, comments, and real-time notifications replace insecure email threads, ensuring confidential and timely exchanges between researchers, organizations, and triage teams.
Fair Reward and Reputation System
Valid reports trigger bounty awards, payout tracking, and automatic reputation updates with leaderboards—addressing the critical lack of incentives and recognition that currently discourages local researchers.
Role-Based Dashboards and Analytics 
Customized views for researchers (submissions, earnings, rankings), organizations (program performance, vulnerability trends), and administrators (platform oversight) .
Secure and Compliant Operations
Built-in security features (encryption, role-based access, audit logs) and integration points for local payment systems (e.g., Telebirr, CBE Birr, bank transfers) ensure safe transactions and regulatory alignment.
　　　　　　Overall Benefits of the Proposed System
By automating and standardizing the entire workflow, the platform directly resolves the core problems of the current manual system
Slow response times → Fast, tracked processing.
Risk of public disclosure → Safe, coordinated handling.
Low participation → Attractive rewards and reputation building.
Inefficiency and errors → Structured, auditable processes.
Dependency on foreign platforms → Affordable, local alternative with data sovereignty.

BEGIN RegisterUser(userDetails, selectedRole)
    IF userDetails.email EXISTS IN Database THEN
        RETURN "Email already registered"
    END IF
    IF userDetails.password NOT meet complexity requirements THEN
        RETURN "Password too weak"
    END IF
    CREATE pendingUser WITH userDetails AND selectedRole
    SET pendingUser.status = "PendingVerification"
    SAVE pendingUser TO Database
    verificationLink = GENERATE unique verification token
    SEND email TO userDetails.email WITH verificationLink
    RETURN "Verification email sent"
END
BEGIN VerifyEmail(token)
    FIND user WITH matching token
    IF user FOUND AND token valid THEN
        SET user.status = "Active"
        REMOVE token
        SAVE user
        RETURN "Account activated successfully"
    ELSE
        RETURN "Invalid or expired link"
    END IF
END

Algorithm for UC-02: Submit Vulnerability Report
BEGIN SubmitVulnerabilityReport(researcherID, programID, reportDetails, attachments)
    IF researcher NOT joined programID THEN
        RETURN "Not eligible for this program"
    END IF
    IF reportDetails missing required fields THEN
        RETURN "Incomplete report"
    END IF
    IF attachments size > 10MB OR count > 5 THEN
        RETURN "Attachment limits exceeded"
    END IF
    CREATE newReport
    SET newReport.ID = GENERATE unique ID
    SET newReport.status = "New"
    SET newReport.submissionDate = CURRENT timestamp
    ASSIGN reportDetails and attachments TO newReport
    SAVE newReport TO Database
    SEND notification TO triage specialists AND program owner
    RETURN "Report submitted successfully - ID: " + newReport.ID
END

Algorithm for UC-03: Validate Report
BEGIN ValidateReport(triageID, reportID, severity, status, comments)
    LOAD report FROM Database USING reportID
    IF report.status != "New" AND report.status != "Triaged" THEN
        RETURN "Report already processed"
    END IF
    // Deduplication check
    similarReports = SEARCH reports WITH similar title/description IN same program
    IF similarReports FOUND THEN
        SET report.status = "Duplicate"
        LINK report TO original similarReport
    ELSE
        SET report.severity = severity (using VRT taxonomy)
        SET report.status = status  // Valid, Invalid, or Triaged
    END IF
    ADD comments TO report
    LOG audit trail (triageID, action, timestamp)
    SAVE report
    SEND notification TO researcher AND organization
    RETURN "Report updated successfully"
END

END

 BEGIN Saga_ApproveReward(reportID, researcherID, amount)
    1. GENERATE unique transactionID (Idempotency Key)
    
    2. CALL WalletService.ReserveFunds(orgID, totalAmount, transactionID)
       IF FAIL: ABORT ("Insufficient funds in organization account")

    3. CALL PlatformService.CreditCommission(commissionAmount, transactionID)
       IF FAIL: 
          CALL WalletService.Compensate_ReleaseFunds(orgID, totalAmount, transactionID)
          RETURN "System Error: Commission processing failed."

    4. CALL ResearcherService.CreditWallet(researcherID, researcherBountyAmount, transactionID)
       IF FAIL:
          CALL PlatformService.Compensate_DebitCommission(transactionID)
          CALL WalletService.Compensate_ReleaseFunds(orgID, totalAmount, transactionID)
          RETURN "System Error: Researcher payout failed."

    5. CALL TriageService.UpdateReportStatus(reportID, "Resolved", transactionID)
       IF FAIL:
          // CRITICAL: If status update fails, we must roll back the money!
          CALL ResearcherService.Compensate_DebitWallet(researcherID, researcherBountyAmount, transactionID)
          CALL PlatformService.Compensate_DebitCommission(transactionID)
          CALL WalletService.Compensate_ReleaseFunds(orgID, totalAmount, transactionID)
          RETURN "System Error: Could not update report status. Funds rolled back."

    6. LOG "Transaction Successful" in AuditTrail
    7. TRIGGER AsynchronousNotification(researcherID, "Payout Complete")
END

The platform is designed to facilitate secure collaboration between ethical hackers (researchers) and organizations that require vulnerability assessment of their applications, systems, and AI models.
　　　　　　　Purpose of the System Design
Provide a robust, scalable, and maintainable architecture.
Ensure performance, reliability, security, and usability for end users.
Support modular growth, enabling addition of features like AI Red Teaming, PTaaS, wSimulation Engine, and BountyMatch.
Ensure clear data persistence, component interaction, and deployment across servers.

The system uses a layered, modular architecture with components organized into presentation, application, domain, integration, and data layers.

　　　　　　　3.4 Access control and security
The Platform implements a Role-Based Access Control (RBAC) system with the principle of least privilege. Each user role has specific permissions aligned with their responsibilities, ensuring data confidentiality, integrity, and system security.
　　　　　　User Roles and Access Levels
I.Researcher (Ethical Hacker)
Access Permissions
Browse public and invited private bug bounty programs
View program details (scope, rules, rewards, disclosure policy)
Submit vulnerability reports with attachments
View own submitted reports and their status
Communicate with triage team via secure messaging
Access bug bounty simulation environment (all levels)
View own earnings, reputation score, and leaderboard ranking
Update own profile and payment information
Collaborate with other researchers on reports (reward splitting)
Participate in live hacking events (if invited)
Accept/Decline BountyMatch recommendations
Access Restrictions
Cannot view other researchers' reports
Cannot access organization dashboards or analytics
Cannot modify program scope or rules
Cannot approve or reject reports
Cannot access financial/payment processing systems
Cannot view platform revenue or commission data
Cannot access admin or triage portals
Cannot modify other users' profiles or data
II.Organization (Program Owner)
Access Permissions
Create and manage bug bounty programs (public/private/VDP)
Define  program scope, rules, reward tiers, and disclosure policy
View all reports submitted to their programs
Communicate with researchers and triage team
Request PTaaS engagements and code reviews
View analytics dashboard (vulnerability trends, severity distribution)
Approve bounty amounts for validated reports
Manage organization team members and roles
Configure SSDLC integrations (Jira, GitHub)
Schedule live hacking events
Create AI Red Teaming engagements
View billing and payment history
Export reports and compliance documentation
Access Restrictions
Cannot view reports from other organizations' programs
Cannot access researcher personal information (beyond public profile)
Cannot modify platform settings or configurations
Cannot access triage queue or validate reports (managed service)
Cannot process payments directly (Finance Officer role)
Cannot access platform-wide analytics or revenue data
Cannot modify other organizations' programs
Cannot access simulation environment backend
III.Triage Specialist (Platform Staff)
Access Permissions
View all submitted vulnerability reports across all programs
Validate and assign severity using VRT taxonomy
Update report status (New → Triaged → Valid/Invalid/Duplicate)
Deduplicate reports and link to original submissions
Communicate with researchers and organizations
Add internal triage notes and comments
Request additional information from researchers
Assign reports to other triage specialists
View triage queue and analytics
Use duplicate checker and triage templates
Recommend bounty amounts based on severity
Access Restrictions
Cannot approve or process payments
Cannot create or modify programs
Cannot access financial data or revenue analytics
Cannot modify platform settings or user roles
Cannot  delete reports (only mark as invalid/duplicate)
Cannot access researcher payment information
Cannot modify organization billing details
Cannot access admin-level system configurations

View all pending payment approvals
Approve  or reject bounty payments

View financial overview and revenue analytics
Calculate and track platform commission (30%)
Generate financial reports and invoices
View transaction history and payment status
Manage payment methods and settings
Export financial data for accounting
View researcher and organization payment details
Handle PTaaS engagement payments
Track bulk payments and processing status
Access Restrictions
Cannot validate or triage vulnerability reports
Cannot create or modify programs
Cannot access report technical details (only payment info)
Cannot modify platform settings or user roles
Cannot access triage queue or researcher communications
Cannot modify severity or report status
Cannot access simulation environment
Cannot perform admin-level system operations

　　　　　　　Security Mechanisms
Authentication
Multi-Factor Authentication (MFA): Required for all roles
Password Policy Minimum 12 characters, complexity requirements
Session Management 30-minute timeout, secure token-based authentication
Email Verification  Mandatory for account activation
Authorization
Role-Based Access Control (RBAC) Enforced at API and UI levels
Least Privilege Principle Users granted minimum necessary permissions
Dynamic Permission Checks Validated on every request
Token-Based Authorization JWT tokens with role claims
Data Protection
Encryption at Rest AES-256 for sensitive data (reports, attachments, payment info)
Encryption in Transit TLS 1.3 for all communications
Data Isolation Organizations cannot access each other's data
Secure File Upload Virus scanning, file type validation, size limits
Audit and Monitoring
Audit Logs All critical actions logged with timestamp, user ID, IP address
Activity Tracking Report submissions, status changes, payment approvals
Security Monitoring Failed login attempts, suspicious activities
Compliance Logging Retention for 90 days minimum
Input Validation
SQL Injection Prevention Prepared statements, parameterized queries
XSS Protection Input sanitization, output encoding
CSRF Protection Anti-CSRF tokens on all forms
Rate Limiting API throttling to prevent abuse
Secure Communication
In-Platform Messaging Encrypted, role-restricted
Email Notifications No sensitive data in emails (links only)
API Security API keys, rate limiting, IP whitelisting (optional)
